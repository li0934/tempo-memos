# Tempo エンタープライズ向け決済の新機能（解説メモ）

出典: [New features on Tempo for enterprise payments](https://tempo.xyz/blog/new-features-for-enterprise-payments)（Tempo Blog、2026年4月27日公開）

プロトコル詳細は公式ドキュメントを正とする。

- **アクセスキー:** [Use Tempo Transactions — Access Keys](https://docs.tempo.xyz/guide/tempo-transaction#access-keys)
- **バーチャルアドレス:** [Use virtual addresses for deposits](https://docs.tempo.xyz/guide/payments/virtual-addresses)

---

## 概要

Tempo は **グローバルなステーブルコイン決済をスケールさせる**ことを目的に設計された **payments-first** ブロックチェーン。本記事は **エンタープライズが日常的に回す決済ワークフローのうち、これまでステーブルコイン上で難しかった部分** を Tempo の新機能で埋める、という位置づけの **プロダクト告知**。

記事が挙げる二つの柱:

| 柱 | プリミティブ | 主な用途 |
|----|-------------|----------|
| **定期・自動・従量課金** | 強化された **アクセスキー**（periodic spending limits など） | サブスク、オートペイ、メーター課金 |
| **入金・台帳・マルチテナント** | **バーチャルアドレス** | 顧客別入金、請求書単位の突合、プラットフォーム口座構造 |

**記事の記載どおり、ここで述べる機能は Tempo メインネットで利用可能。** ユースケースのスコープ設計やパイロットは [Stablecoin Advisory](https://tempo.xyz/advisory) が案内している。

---

## 背景：ステーブルコインでまだ埋まっていなかったギャップ

一度きりのグローバル決済はステーブルコインインフラで実用的になりつつある一方、エンタープライズ運用の **残りの半分** — **定期フロー**、**入金の帰属（アトリビューション）**、**マルチテナントなウォレット構造** — は依然として難しかった、という問題設定。

本記事は Stablecoin Advisory の立ち上げと、Tempo 上で本番稼働するエンタープライズの紹介（前週）の文脈で書かれている。

---

## 1. 強化されたアクセスキーによる定期決済

### 定期課金の選択肢 — ブログの「二択」だけでは足りない

[enterprise payments のブログ](https://tempo.xyz/blog/new-features-for-enterprise-payments) は、ステーブルコイン定期課金の問題を **「都度承認」か「過剰な一括承認」** の二択として短く書いている（[docs/subscriptions.md](subscriptions.md) も同趣旨の整理）。実務では **第三の定番** もある。読者がここで止まると「Tempo なくても `approve` で足りるのでは？」となりやすいので、先に並べてから Tempo の答えを述べる。

#### 1. 課金のたびにユーザーへ再プロンプト

毎回ウォレット署名・決済 UI。サブスクとして摩擦が大きく、解約・失敗時の UX も重い。

#### 2. 実際の必要額より遥かに大きな枠を一度だけ承認

例: 年間分をまとめて許可。都度署名は避けられるが **過剰許可** になり、利用者リスク・コンプラ説明が難しくなる。

#### 3. ERC-20 の `approve`（＋請求コントラクトの `transferFrom`）

**EVM 上の定番。** ユーザーが請求用コントラクト（spender）にトークン allowance を渡し、サービスがスケジュールで `transferFrom` して pull する。

```
ユーザー → billing コントラクトに allowance を approve
サービス → 毎月 transferFrom(user, merchant, 定額)
```

**できること:** 都度ウォレット署名なしの定期引き落とし（サブスク請求コントラクトパターン）。

**`approve` との決定的な差 — 期間上限を誰が守るか**

ここが一番の違い。

`approve` では、ユーザーが渡すのは **「このコントラクトが、合計いくらまで引けるか」** という **一つの枠** だ。例えば年間 $2,600 分をまとめて approve したら、請求側のコントラクトは **初回の `transferFrom` で $2,600 全部引き落とすことも、プロトコル上は可能** だ。「週 $50 ずつ 52 回」という **運用の約束** を守るかどうかは、**請求コントラクトの実装（＝相手を信頼するか）** に委ねられる。

アクセスキーでは **「週あたり最大 $50」** のように **期間ごとの上限** をプロトコルが持つ（[TIP-1011](https://docs.tempo.xyz/protocol/tips/tip-1011) の periodic limits）。請求は **その期間の残り枠を超えたらそもそも通らない**。だから **1 回目に上限まで一括引き落とす** ことは防げる。期間が終われば枠は **自動でリセット** される。

具体例:

| | `approve`（年 $2,600 を一括許可） | アクセスキー（週 $50 × 52 週） |
|--|-----------------------------------|-------------------------------|
| 1 回目に取れる最大 | **$2,600 全部**（コントラクト次第） | **$50 まで**（プロトコルが拒否） |
| 52 週の上限を守るには | 請求側が **善意の実装** であること | **プロトコルが強制** |

`approve` で週 $50 だけ許可し続ける方法もあるが、それは **毎週 approve し直す** か、**枯れたらまた署名** の世界に戻り、サブスクとしては 1 と同じ摩擦が出やすい。

**そのほかの差（短く）**

- **スコープ:** `approve` は「このトークンをこのコントラクトが引ける」まで。アクセスキーは **どのコントラクトのどの関数か** までプロトコルが縛れる（[docs/access-keys.md](access-keys.md)）。
- **渡すもの:** `approve` は引き出し許可。アクセスキーは **メイン鍵を渡さない** サブ鍵（エージェント・デバイス向け）。
- **基盤:** アクセスキーは Tempo Transaction 上のネイティブ機能（fee sponsorship などと一体）。

**まとめ:** 「定期 pull 課金ができるか」だけ見れば `approve` でも近い。**期間ごとの上限をプロトコルが強制する** 点が、アクセスキーの実務上の価値の中心。

### アクセスキーで何が変わるか（記事の主張を補足）

**Enhanced access keys** は、上記 1〜3 の **どれか一つに置き換わる** というより、**3 を超えるプロトコルネイティブな委譲** として位置づけると理解しやすい。

- ユーザーは **期間あたりの上限** を **一度だけ** 承認（例: **52 週間、週あたり最大 $50**）— 2 の「一括過剰」と 3 の「枯れたら再 approve」の間を避ける
- サービスはその上限内で **スケジュールに沿って請求（pull）**
- **期間が終わると上限は自動リセット**
- ユーザーは **いつでも承認を取り消し可能**
- キーは **特定コントラクト・関数にスコープ** — 3 より **プロトコル層で** 別用途への流用を抑える

詳細は [docs/access-keys.md](access-keys.md) および [docs/subscriptions.md](subscriptions.md)（Subscriptions ブログ要約）を参照。

### ユースケース（記事）

#### サブスクリプション課金

SaaS・ストリーミングなど **定額の定期課金**。購読者がアクセスキーを一度承認すれば、事業者は **国ごとの決済インフラを揃えず** 同じスケジュールで各国から請求でき、**即時決済** になる、というメッセージ。

関連ブログ: [Subscriptions on Tempo](https://tempo.xyz/blog/subscriptions-on-tempo)

#### オートペイ（自動支払い）

**定期の支払義務** — 公共料金、家賃、ローン servicing など — をスケジュール支払いに使える。記事は **従来の ACH デビットに近いパターン** と対比しつつ、

- 決済は **即時**
- 承認は **特定の受取人（payee）にプログラム的にスコープ**
- ユーザーは **受取人に連絡せずに取り消し可能**

と述べている。

#### メーター・従量課金

**期間ごとの上限付きで、実使用量に応じて請求** するモデル。例: 「今月最大 $500 まで」と承認し、API コール数・コンピュート時間・データアクセス・スループットなど **実績ベース** で請求。

- **上限** が顧客を保護
- **定期承認** がサービス側オペレーションを簡素化
- 双方が **プリペイドクレジット管理** や **超過サプライズ** を避けやすい、という整理

高頻度・低単価の従量（トークン単位など）の別パターンとして **MPP Sessions** も関連（[docs/mpp-sessions.md](mpp-sessions.md)）。**アクセスキーと MPP Sessions は別物** — 下記「アクセスキー vs MPP Sessions」参照。

#### アクセスキー vs MPP Sessions — 混同しやすい点

どちらも「一度許可してあとは自動で払う」ように見えるが、**解いている問題が違う**。

| | **アクセスキー** | **MPP Sessions** |
|--|-----------------|------------------|
| **何の仕組みか** | Tempo 上の **署名権限委譲**（サブ鍵 + 上限・スコープ） | MPP の **ペイメントチャネル**（escrow + オフチェーン voucher） |
| **向く課金** | **期間ごとの上限**内での pull（サブスク・オートペイ・月次従量 cap） | **リアルタイム・超高頻度**の従量（トークン/バイト/API コール単位） |
| **請求のタイミング** | サービスが **スケジュールや請求サイクル** で pull | 利用と **同時**（ストリーム中に voucher を差し替え） |
| **オンチェーン** | pull のたびに **Tempo トランザクション**（ユーザー再署名なし） | **open + close** の 2 回に中間を集約（voucher はオフチェーン） |
| **上限の考え方** | 「**週 $50**」「**月 $500**」など **カレンダー／固定長ウィンドウ** | escrow に預けた **デポジット上限**（枯れたら [top up](https://mpp.dev/payment-methods/tempo/session)） |
| **典型例** | Netflix 月額、家賃の自動引き落とし | LLM トークンストリーム、秒間数千 tick のデータフィード |

**組み合わせ・signer の渡し方** など詳細は **[docs/mpp-sessions.md — アクセスキーとの関係](mpp-sessions.md)** および同ファイル **「現状どう使うか（クライアント）」** 節を参照。

**読み分け（要約）:** **請求サイクルと期間 cap** → アクセスキー。**利用中の超高頻度従量** → MPP Sessions。両方併用もある。

## 2. バーチャルアドレスによる入金・プラットフォームウォレット構造

### 典型的な問題

**顧客資金を預かる事業** では、顧客ごとに **専用の入金アドレス** を渡すと突合しやすいが、多くのチェーンでは **各入金先が実オンチェーン口座** になり、

- 初期化
- 監視
- トレジャリーへの **スイープ**（別トランザクション）

が必要で、**スケールすると運用コストが prohibitive** になる、という問題設定（[docs/virtual-addresses.md](virtual-addresses.md) と同根）。

### 記事の説明（仕組み）

**Virtual addresses** がプロトコル層で解く、という整理。

1. 事業者が **マスターウォレットを一度登録**
2. **オフラインで** 顧客（や請求）ごとの **一意のバーチャルアドレスを派生**
3. いずれかへ送金された TIP-20 は **スイープなしでマスターへ直接クレジット**
4. **アドレスごとの初期化コストなし**
5. 送信者には依然 **一意の送金先**、事業者側は **一本の集約ウォレット**

### ユースケース（記事）

#### 顧客別入金の帰属

取引所・ネオバンク・ランプ・カストディアン向け。**顧客ごと専用入金先** を維持しつつ、着金は **自動で帰属**、オペレーターは **数千ウォレットを個別管理するインフラ税** を避ける、という説明。

#### 請求書（インボイス）単位の突合

**顧客 1 人 1 アドレス** ではなく **請求書 1 件 1 アドレス** も可能。入金が到着した時点で **インボイスに自動マッチ** し、財務チームが送信者と明細を手作業で突合する負荷を減らす、という整理。

#### マルチテナント・ウォレット構造

フィンテック・組込み金融向け。**スポンサー銀行のバーチャル口座**（マスター口座 + エンドユーザー別サブ口座・残高管理）の **オンチェーン analogue** として位置づけ。

- プラットフォームは **マスターウォレット 1 本**
- 各エンドユーザーに **専用バーチャルアドレス**
- **顧客レベルの構造** を保ちつつ、**ユーザーごとに別アカウントを持つコスト** を避ける

### 運用上の注意（記事）

- **マスターウォレットのアドレスは送信者に直接露出しない** — 顧客は自分のバーチャルアドレスのみとやり取り
- **TIP-20 ステーブルコイン向け** — **TIP-20 以外** をバーチャルアドレスへ送ると **失われる可能性** がある（記事の警告）

関連ブログ: [Virtual addresses for stablecoin deposits](https://tempo.xyz/blog/virtual-addresses)、[TIP-20](https://tempo.xyz/blog/tip20)

## 二機能の関係（読み方）

| 方向 | アクセスキー | バーチャルアドレス |
|------|-------------|-------------------|
| 資金の流れ | ユーザー口座からサービスが **pull（引き落とし）** | 外部からプラットフォームへ **push（入金）** |
| 典型 | サブスク・オートペイ・従量請求 | 入金帰属・請求書突合・マルチテナント |
| 詳細メモ | [access-keys.md](access-keys.md)、[subscriptions.md](subscriptions.md) | [virtual-addresses.md](virtual-addresses.md) |

エンタープライズの多くの台帳は **入金（push）と定期請求（pull）の両方** を持つ。本記事はその両輪を **メインネットで同時に使える** と宣言している。
