# Tempo のバーチャルアドレス（ステーブルコイン入金向け）解説メモ

出典: [Virtual addresses for stablecoin deposits](https://tempo.xyz/blog/virtual-addresses)（Tempo Blog、2026年4月27日公開）

プロトコル詳細は **[TIP-20 のバーチャルアドレス](https://docs.tempo.xyz/protocol/tip20/virtual-addresses)**（公式ドキュメント）および関連 TIP を正とする。

---

## 概要

Tempo の **バーチャルアドレス（virtual addresses）** は、取引所・オンプラン／オフランプ・ネオバンク・金融機関などが **顧客ごとに異なる入金先アドレス** を発行しやすくするための仕組み。**スイープ（集約送金）トランザクションのオーバーヘッドなし**に、送金先として一意のアドレスを持たせられる、というのが記事のメッセージ。

バーチャルアドレスへ送られた TIP-20 ステーブルコインは、プロトコル層で **登録済みのマスターウォレットに直接計上（クレジット）**される（記事本文）。

評価やパイロットの相談は記事がエンタープライズチームへのコンタクトを案内している。

---

## 典型的な「入金アドレス」の問題

決済事業では「受取人ごとに別アドレス」を出すと **突合（レコンサイル）が楽**になる一方、スケールするとコストが載りやすい、という問題設定が記事で語られている。

### オンチェーンでよくある負担

- **資金がそのデポジットアドレス自体に滞留する** — チェーンによっては口座を受け取り可能にするために **状態賃料（state rent）のようなコスト** がかかり、規模が出ると割に合わない、との記載がある。
- **トレジャリーへ集約したいならスイープが必要** — 別トランザクションで振り出すため **latency・ガス・失敗モード** が増える。
- **監視・突合のインフラが重くなる** — サポート・財務・オペレーション・セキュリティ・コンプラなどが「どこに着金したか」「いつ動いたか」を追う負荷が上がる。
- **台帳設計・トレジャリー運用・「受領」と「利用可能」の間の詰まり**など、エンタープライズ向けオーバーヘッドとして繰り返し触れられている。

## バーチャルアドレスの考え方（記事の説明）

記事の要点は次のような対比になっている。

| 通常イメージ | Tempo のバーチャルアドレス |
|--------------|---------------------------|
| 入金先アドレス＝実アカウントとして残高を抱える | **見た目は普通のアドレスだが、単独ウォレットとして資金を抱え続ける前提ではない** |
| 後からトレジャリーへ送る（スイープ） | **TIP-20 の転送としてプロトコルが解決し、マスターへ直接クレジット** |

運用イメージ:

1. 事業者が **マスターウォレットを一度登録**する。
2. その後、**オフラインで大量の顧客専用バーチャルアドレスを派生**できる（記事では「trillions」と表現）。
3. 送信者がそのバーチャルアドレスへ TIP-20 を送ると、Tempo が形式を検知し **登録マスターへ解決して計上**する。
4. **送信者側には依然として一意のアドレスがある**一方、**オペレーター側はスイープ無しで一本の実ウォレットに集約**できる、という整理。

### マスターからバーチャルを派生させるコードはあるか？

**ある。** ただし公式が載せている API は **`master` の EOA アドレスだけで一行っていう形ではなく、`masterId` と `userTag` を渡す**ものになる。

流れは次の二段になる。

1. **オンチェーンで一度、`virtual master` を登録**する（ドキュメントでは salt をマイニングして `masterId` を得る流れが説明されている — `VirtualMaster.mineSaltAsync` など。[Use virtual addresses for deposits](https://docs.tempo.xyz/guide/payments/virtual-addresses) 参照）。
2. **登録後はオフラインで**、`masterId` と事業者が決める **`userTag`（顧客・口座・請求などのルーティング用 6 バイト）**からバーチャルアドレスを計算する。

公式ガイドのコード例（ライブラリ **ox/tempo**）:

```typescript
import { VirtualAddress } from "ox/tempo";

const virtualAddress = VirtualAddress.from({
  masterId,
  userTag: "0x000000000001",
});
```

`userTag` は運用上「内部の顧客 ID に相当する値」を載せるイメージで、[TIP-1022](https://docs.tempo.xyz/protocol/tips/tip-1022) では **プロトコルは内容を解釈せず不透明**とされている。

**まとめ:** 「マスターアドレス → ハッシュでバーチャル」とだけ読むより、**マスターアドレスがレジストリに登録されて初めて確定する `masterId`** がキーで、その **`masterId` + `userTag` が 20 バイトのアドレス形式にエンコードされる**、と捉えると公式・TIP と整合しやすい。

### 先頭 4 バイトが `masterId` で、登録者が増えると破綻しないか？

**はい、バーチャルアドレスの先頭 4 バイトが `masterId`（続く 10 バイトがマジック、`userTag` が末尾 6 バイト）というレイアウトは [TIP-1022](https://docs.tempo.xyz/protocol/tips/tip-1022) の通り。**
ただし **`masterId` は「世界中で共通の 1 個」ではなく**、**各 `registerVirtualMaster` がレジストリに載せる識別子**として振られる。

- **事業者やマスターが複数いれば**、通常は **それぞれ別の `masterId`** が対応する（レジストリは `masterId → マスターのアドレス`）。「利用者が増えたらみんな同じ先頭 4 バイトになる」というモデルではない。
- **4 バイトなので理論上コリジョンはありうる**。TIP では **別の `(masterAddress, salt)` が同じ `masterId` に当たったら `MasterIdCollision` でリバートし、別 salt を試す**と書かれている。登録には **salt のプルーフ・オブ・ワークー**もあり、設計意図が説明されている。
- したがって「全員がマスター登録すると破綻」のイメージより、**名前空間はレジストリとコリジョン処理で運用される**のが仕様に近い。

### 「そんなにマスター登録しない」「Tempo が審査する」のでは？

**オンチェーンのプロトコルとして読める範囲では、[TIP-1022](https://docs.tempo.xyz/protocol/tips/tip-1022) の `registerVirtualMaster` は「Tempo が書類を見て承認したらレジストリに載る」というモデルには読みにくい。** 関数は **`external`** で、`msg.sender` をそのマスターアドレスとして **`salt` が PoW を満たし、`masterId` が未占有なら登録**、という記述になっている。

リバートが説明されているのは例として **`ProofOfWorkFailed`**（PoW 不一致）、**`MasterIdCollision`**（`masterId` の取り合い）、**`InvalidMasterAddress`** のような **技術的条件**である。**「有効なマスターアドレス」**とは仕様上、`address(0)` でなく、バーチャル形式そのものではなく、一定プレフィックスの TIP-20 トークンコントラクトのアドレスでないこと、などが挙げられている（[TIP-1022 Registry の節](https://docs.tempo.xyz/protocol/tips/tip-1022)）。

だから **利用実態として「エンドユーザーがみんなマスターを登録する」わけではない**（入金インフラを張る事業者側が登録する想定）は読んでいて筋が通る一方、**「プロトコルレベルで人数が限られる／会社ごとの許認可が必須」**とは仕様からは読み取れない。**製品・営業・コンプラや提携について問うエンタープライズ連絡先**は別レイヤー（ブログや advisory の話）であり、**チェーン上の登録そのものを Tempo が逐一審査する必然とは結べない**。

なお **転送時は `effectiveRecipient`（解決後のマスター）に対して従来どおり TIP-403 などの受領ポリシー**が効く、という説明があり（転送パスの説明）、**マスターがチェーンの規則上そのトークンを受け取れないなら入金はリバートしうる**が、それは **事前「登録審査」というよりポリシー検証**の話になる。

## 向くユースケース（記事の列挙）

- 取引所・ブローカーが **入金用に専用アドレスを発行**する
- オンプラン／オフランプが **インバウンドを顧客に紐づける**
- マーケットプレイスが **加盟店や請求ごとにアドレスを分ける**
- トレジャリープラットフォームが **セグメント別の入金を受ける**

共通ニーズは記事の言い方では **「誰の支払いかを正確に帰属させつつ、滞留資産は実質ワンアドレスで管理したい」** に近い。

## エンタープライズ視点でのメリット（記事の見出し）

記事は「アプリ層の転送ハックではなく、**TIP-20 の転送の扱いに組み込まれている**」というニュアンスで説明している。

- **顧客・フロー単位の帰属（アトリビューション）** — バーチャルアドレスが顧客・通路・フローを表しうる。
- **スイープ操作が不要** — 二次トランザクションでデポジットアドレスから逃がす運用を削減。
- **インフラ負荷の抑制** — **エンドポイントごとに別オンチェーンアカウントを増やさない**（記事では新規アカウントコストの話にも触れている）。

また **銀行がフィンテックにバーチャル口座を発行させる**関係に近い **オンチェーンのアナロジー**として紹介されている。

## よくある疑問：金融機関は顧客ごとに口座を厳密管理では？ / なぜワンアドレス集約と言える？

**いちばんのポイントは「どのレイヤで厳密か」の話**になる。

従来の銀行でも、顧客に見えている **口座番号・残高一覧の厳密さ**（誰にいくら負債として計上するか）は **勘定系・台帳** で担保される。一方で、実際のコレスポンデント銀行預けや **実名の保管口座（オムニバス）** は **一本にまとめて運用**し、顧客ごとの残高は **内部のブック上の分割**で追う、という形はフィンテックでもよくある（記事が触れる **バーチャル口座番号** に近い）。

バーチャルアドレスも **オンチェーンのトークン残高が物理的にどのアドレスにあるか** と **誰の入金として帰属させるか** を切り離して説明している。

- **プロトコル／オンチェーン**: 登録済み **マスターウォレット** にステーブルコインが載る（記事の「直接クレジット」）。
- **事業者側の厳密な顧客別管理**: **どのバーチャルアドレス宛てにいくら着いたか** をイベントやインデクサで追い、**自社の台帳・与信・コンプラフローに紐づける** — これがブログでいう **レコンサイル（突合）のための一意エンドポイント**と対になる。

だから「金融機関は顧客別に厳密」と「オンチェーンではワンアドレスに見える」は **矛盾しない**。厳密さは **内部制度・台帳・帰属ロジック** にあり、オンチェーンは **スイープや多数の実アカウントを増やさずに入金を届ける輸送路**として設計されている、という読み方になる（実務・規制の判断は各社のコンプラ・法務が管轄）。

### バーチャルアドレスで「署名」できる？（送金元として使える？）

**このプロトコルでのバーチャルアドレスは、別ウォレットではなく入金用のエイリアス**である。[Virtual addresses for TIP-20 deposits](https://docs.tempo.xyz/protocol/tip20/virtual-addresses) では「A virtual address is **not a second wallet**. It is a **deposit alias** for one canonical wallet」とあり、トークン残高は **`balanceOf(virtualAddress)` は常に 0** で、実体はマスターウォレット側、と明示されている。

よって **「バーチャルアドレスに対応する秘密鍵を持って、いつもの EOA のようにトランザクションに署名する」というモデルではない**。署名できるのは **鍵を実際に握っているアカウント**（マスターやユーザーのスマートアカウントなど）であり、バーチャルアドレスは **受け取り先・帰属トレース用の 20 バイトの住所形式**（TIP-1022 のレイアウト）として設計されている。

また公式は **[TIP-1022 の対象は限定的](https://docs.tempo.xyz/protocol/tip20/virtual-addresses)** であり、「あらゆるプロトコルがバーチャルアドレスを解決するわけではない」「汎用の EVM エイリアスでもない」と書いている。**借り物のアドレスで署名して認証する仕組み**ではなく、**カバーされた TIP-20 の転送・ミント経路での転送先ルーティング**が中心。

### 出金・払い出しは公式ドキュメントに載っているか？

**バーチャルアドレス機能としては「載っていない」が、それは仕様の隙間というよりスコープの切り方**に近い。

公式ガイドの見出しは例として **[Use virtual addresses for deposits](https://docs.tempo.xyz/guide/payments/virtual-addresses)**（**入金向け**）であり、説明の中心は「外部からバーチャル宛に送る → マスターにクレジットされる」まで。プロトコル詳細ページも **[Virtual addresses for TIP-20 deposits](https://docs.tempo.xyz/protocol/tip20/virtual-addresses)** と **deposit** が名前に入っている。

**出金・ユーザーへの払い出しの論理**は、ドキュメントが暗に前提にしているのは次のような整理になる。

1. **残高はマスターウォレット（等）に実体がある** — 入金後のトークンはそこにある。
2. **顧客へチェーン上で送り返す／外部アドレスへ出す**ときは、**通常の TIP-20 送金**（マスターまたはホットウォレットからの `transfer` など）で行う。これは **バーチャルアドレスの転送先解決とは別問題**であり、取引所・決済事業者がもともとやっている **トレジャリー／オペレーション設計**の領域。
3. **社内台帳**では、入金時にバーチャルアドレス＋ `userTag` で顧客にクレジットし、出金時に内部残高をデビットしてからオンチェーン送金を発行する、という二段構成になりうる。

つまり **「バーチャルアドレスから出金する」のではなく「マスター側で資産を把握したうえで送金する」**が実運用モデルになり、**プロトコルが決めているのは入金経路のルーティング**である。

[ブログ末尾](https://tempo.xyz/blog/virtual-addresses) は deposit と並んで merchant settlement や **payout infrastructure** にも触れるが、それらは **より広い決済スタック**の話であって、TIP-1022 が「出金専用の仕様」を追加するという読みではない。**運用設計や統合はエンタープライズ向けドキュメント・問い合わせ先に依存する**、という整理になる。

### 出金時、`Transfer` の `from` はバーチャルアドレスになる？

**基本はならない。** ユーザー・ビジネスが「バーチャル宛と取引していた」のと同じ見せ方を、**オンチェーンの送金元として自動では再現しない**のが仕様に沿った読み方になる。

[TIP-1022](https://docs.tempo.xyz/protocol/tips/tip-1022) は **TIP-20 の転送先（recipient／`to` がバーチャル形式）をどうマスターに解決するか** が中心であり、送り手側には適用しない、という書きぶりがある。

公式でも **「The `from` address on `transferFrom`, `transferFromWithMemo`, and `systemTransferFrom` is not affected by TIP-1022 resolution.」** とあり、また **`to` 側の解決**と **トランザクション送信者**は別だという趣旨が説明されている。

その結果、**マスターが署名してユーザーの外部アドレスへ払い出す**ときは、通常 **`Transfer` の送信側はマスター（またはホットウォレット等・実際に署名しているアカウント）**になり、**バーチャルアドレスが `from` としてイベントに現れるモデルは、TIP-1022 が用意しているデフォルトではない**。

入金側では **`Transfer(sender, virtualAddress, …)` がイベントに残る**ので「そのバーチャル宛に誰かが送った」帰属は取れる。**出金側**で相手に「どの顧客分か」をチェーンだけで見せたければ、**台帳・オフチェーン・別フィールド**での説明になりやすい（詳細は事業フローとコンプラ）。

## 公式ドキュメント・製品との関係

このブログは製品・運用のストーリー寄り。**イベントの並びやマスターウォレットへの論理入金の扱い**などは、[Virtual addresses for TIP-20 deposits](https://docs.tempo.xyz/protocol/tip20/virtual-addresses) のような **プロトコル／開発者向け文書**と突き合わせること。

T3 アップグレードの説明では、バーチャルアドレス入金が **同一トランザクション内で二つの `Transfer` イベント**を出すなどの **インデクサ／エクスプローラ向け注意**も書かれている（[T3 Network Upgrade](https://docs.tempo.xyz/protocol/upgrades/t3) の「Virtual addresses for explorers and indexers」など）。ブログ本文だけではないレイヤの話なので、統合時はそちらも参照する。
