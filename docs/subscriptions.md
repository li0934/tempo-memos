# Tempo におけるサブスクリプション（解説メモ）

出典: [Subscriptions on Tempo](https://tempo.xyz/blog/subscriptions-on-tempo)（Tempo Blog、2026年4月27日公開）

---

## 概要

Tempo の **アクセスキー（access keys）** が **定期的な支出上限（periodic spending limits）** に対応したことで、**ユーザーへ何度も署名を求めず**、かつ **ユーザーの鍵を預けずに**、オンチェーンでサブスクリプション課金ができるようになった、という内容。

人間の利用者向けと **AI エージェント向け**の両方で同じ仕組みが使える。ステーブルコインで定期・スコープ付き課金フローを組む場合は `partners@tempo.xyz` に問い合わせ可能（公式記事より）。

---

## 背景：定期課金とステーブルコインのギャップ

多くのビジネスは「毎週・毎月・毎年の定額」という **定期課金** が中心。クレジットカードならこのモデルは成立しやすいが、**カードを持たない顧客**や、**ステーブルコイン主体で動く AI エージェント**には向きにくい。

ステーブルコインは **一度きりのグローバル決済**には強い一方、**繰り返し課金**は難しかった。代表的な妥協案は次の二つで、どちらも UX・リスクの面で問題がある。

1. **課金のたびに都度ユーザーの承認**を取る（承認のたびに摩擦が増える）
2. **すぐ必要な額より遥かに大きな枠**だけ一度まとめて承認させる（例：年分まとめて）。過剰な許可になり、利用者リスクも高まる

Tempo はこの問題を、アクセスキーと **期間ごとにリセットされる上限** の組み合わせで切り分ける。

## アクセスキーが Tempo でどう機能するか（記事の説明）

アクセスキーは Tempo 上のネイティブなプリミティブで、さまざまな決済ユースケースの土台になる。

1. ユーザーがアクセスキーを **最大額** と **有効期限** を指定して承認する
2. **アカウント全体ではなく**、そのキーだけをサービスに渡せる（サービスに「フルコントロール」を渡さない）
3. サービスは、その上限と期限内でユーザー口座から資金を **「引き出す（pull）」** ためにキーを使える
4. キーは **特定のコントラクトや関数セレクタにスコープ**できる（下記「スコープの意味」参照）
5. ユーザーは **いつでも承認を取り消せる**

### スコープの意味（「別用途」をどう防ぐか）

「用途」を **意味として判断しているわけではない**。引き落とし（トランザクション）のたびに、プロトコルが **そのキーで許可されている条件** に **呼び出し内容が合うか** を検証する。

記事の説明どおり、キーは **特定のコントラクトアドレス** と **関数セレクタ**（どの関数を呼ぶか）などに **バインド**できる。許可されていないコントラクトへの呼び出し・別の関数への呼び出しは、プロトコル上 **そもそも成立しない**（＝「そのキーを別のサービスや別操作に流用する」ことが防げる）。

だから「支払いの瞬間に用途を推論する」のではなく、**設計時にユーザーが許した宛先・操作だけに絞った署名権限**になっている、という理解になる。細部の検証ルールは公式のアクセスキー仕様で確認すること。

## 周期性の支出上限でサブスクが成立する理由

**定期的な支出上限**により、その承認は **期間ごとの上限が自動でリセットされる**形にもできる。

- 例：**「50週間、毎週最大50ドル」** のような **ランプサム一括**ではなく、**期間ごとのキャップ**として表現できる（※脚注参照）
- プロトコルは **その期間の残り枠** に対して各請求を検証する。収まれば成功、収まらなければ失敗
- **期間が終わるとウィンドウがリセット**される。ユーザーは条件を一度決めるだけで、その後はサービス側がスケジュールに沿って課金でき、**再プロンプトなし**で運用できる

## 何が可能になるか（記事の例）

- **複数国に SaaS で課金する会社**: 国ごとの決済インフラをすべて揃えなくても、購読者はステーブルコインで **同じフロー**（アクセスキー一度の承認 → 定期請求 → 即時決済）になりうる
- **有料推論 API を使う AI エージェント**: その API のコントラクトにスコープされたキーと、例として **月100ドル上限** などを付与。エージェントは期間中、そのキーで支払いを認証し、各請求は残枠に対して検証され、上限は自動リセット。**同じキーで別コントラクト／別関数への呼び出しはスコープ外なので（上記）、記事の「ほかに使えない」という意味**
- **ニュースレターやストリーミングなど定額サービス**: 定額に合わせた上限のアクセスキーでサブスクとして組める

## 脚注（記事原文）

定期的な上限は **現時点では固定長の期間**（例：1日、1週間）のみ。**カレンダー月のような単位**は、**将来のアップグレード後**に利用可能になる予定（公式記事脚注より）。

### 解釈の注意（いわゆる「月額」との関係）

ここでいう「後からカレンダー対応」とは、主に **請求サイクルをカレンダー境界に揃える**話（例：**毎月1日に締めてリセット**、暦に沿った「1か月」）が **まだプロトコル上そういう単位では指定できない**、という意味に読める。

だから **「カレンダー月ベースの月額プラン」そのものを、そのままプロトコルの periodic limit で表現するのは、現時点では脚注どおり未対応**に近い。

一方で、脚注は **固定長（例：日・週）** と **カレンダー単位**の対比であり、「約30日ごと」など **別の固定長が追加で許されているか**は、このブログ脚注だけでは断定しない方がよい。**実際に組みたい課金粒度は公式のアクセスキー／periodic limits の現行仕様で確認する**こと。
